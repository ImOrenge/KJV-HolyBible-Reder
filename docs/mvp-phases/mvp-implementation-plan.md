# MVP 전체 구현 계획

이 문서는 PRD 기반 MVP를 실제 구현 순서로 정리한 실행 계획이다. 현재 전제는 "기능 검증 가능한 v0 MVP"이며, 실제 OAuth와 성경전서 전체 DB 삽입은 후속 통합 단계로 분리한다.

## 현재 구현 전제

- 계정 OAuth는 실제 provider 연동 없이 mock으로 둔다.
- 앱에는 로그인 없이 진입하는 `guest-reader` local session과 계정 로그인 사용자를 모두 지원한다.
- 로그인/로그아웃 UI는 만들되 실제 OAuth callback, token refresh, provider 계정 연결은 구현하지 않는다.
- 성경전서 전체 본문 삽입은 DB 구현 단계 이후로 미룬다.
- 초기 리더는 성경 권/장 메타데이터와 소량의 fixture 본문으로 구현한다.
- 본문 데이터 접근은 나중에 DB로 교체하기 쉽도록 repository/interface 뒤에 둔다.
- 개인 데이터는 초기에는 local/mock storage로 동작하게 만들고, DB 단계에서 같은 도메인 모델로 옮긴다.

## MVP v0 완료 정의

- 사용자는 로그인 없이 성경 리더에 진입할 수 있다.
- 로그인 사용자는 계정 기준 localStorage 기록으로 이어 읽을 수 있다.
- 사용자는 성경 권/장/절 리더 UI를 사용할 수 있다.
- 사용자는 fixture 본문 기준으로 이어 읽기, 읽음 완료, 통독률 흐름을 확인할 수 있다.
- 사용자는 구절 강조, 인용 구절 저장, 태그, 복사를 사용할 수 있다.
- 사용자는 현재 장 또는 선택 구절을 브라우저 TTS로 들을 수 있다.
- 사용자는 글자 크기, 줄 간격, 다크모드를 조정할 수 있다.
- 실제 OAuth, 운영 DB, 성경전서 전체 seed, 라이선스 확정은 별도 통합 게이트로 남긴다.

## 구현 아키텍처 방향

### 인증

- 비로그인 사용자는 `guest-reader` local session으로 리더를 사용한다.
- 로그인 사용자는 인증 user id 기준으로 localStorage 개인 데이터를 분리한다.
- legacy mock 데이터가 있으면 로그인 계정으로 가져올 수 있게 한다.
- 필요하면 개발 편의를 위해 mock 사용자 전환 기능을 둔다.
- UI 용어는 "로그인"을 유지하되, 비로그인 리더와 계정 기반 기록 경계를 코드와 문서에 명시한다.
- 실제 OAuth로 전환할 때 필요한 인터페이스는 미리 분리한다.

### 성경 데이터

- `BibleRepository` 또는 동등한 데이터 접근 계층을 둔다.
- v0에서는 `books`, `chapters`, `verses` fixture를 반환한다.
- 성경 66권 메타데이터는 먼저 넣을 수 있다.
- 전체 구절 본문은 DB 구현과 라이선스 확인 이후 seed script로 넣는다.
- UI는 fixture가 일부 장만 있어도 깨지지 않게 "본문 준비 중" 상태를 처리한다.

### 개인 데이터

- 읽기 위치, 완료 장, 강조, 인용 구절, 태그, 설정은 도메인 타입을 먼저 정의한다.
- v0 저장소는 localStorage 또는 mock API를 사용한다.
- 저장소 구현은 `UserDataRepository` 같은 인터페이스 뒤에 둔다.
- DB 전환 시 UI 컴포넌트를 크게 바꾸지 않는 구조를 목표로 한다.

## 실행 단계

## Step 0: 프로젝트 골격과 도메인 계약

### 목표

구현 전반에서 바뀌지 않을 타입, 라우팅, mock 데이터 접근 방식을 먼저 잡는다.

### 체크리스트

- [ ] Next.js, TypeScript, Tailwind 기반 프로젝트를 생성한다.
- [ ] 앱 라우트 구조를 정한다.
- [ ] `Book`, `Verse`, `ReadingProgress`, `CompletedChapter`, `Highlight`, `FavoriteVerse`, `Tag`, `UserSettings` 타입을 정의한다.
- [ ] `AuthProvider` mock 계약을 만든다.
- [ ] `BibleRepository` 계약을 만든다.
- [ ] `UserDataRepository` 계약을 만든다.
- [ ] fixture 데이터 폴더를 만든다.
- [ ] 성경 66권 메타데이터 fixture를 만든다.
- [ ] 소량의 장/절 본문 fixture를 만든다.

### 완료 기준

- 앱이 실행된다.
- mock 사용자와 fixture 성경 데이터를 코드에서 조회할 수 있다.
- 이후 기능이 같은 도메인 타입을 공유한다.

## Step 1: 앱 Shell, Mock 로그인, 기본 리더

### 목표

사용자가 로그인 없이 리더에 진입하고 권/장/절 UI를 탐색할 수 있게 한다. 계정 로그인은 기록을 계정 기준으로 이어가기 위한 선택 흐름으로 둔다.

### 체크리스트

- [ ] 홈/대시보드 shell을 만든다.
- [ ] 비로그인 리더 진입과 로그인/로그아웃 UI를 만든다.
- [ ] 성경 읽기 화면을 만든다.
- [ ] 권 선택 UI를 만든다.
- [ ] 장 선택 UI를 만든다.
- [ ] 절 단위 본문 렌더링을 만든다.
- [ ] fixture에 없는 장은 본문 준비 중 상태로 처리한다.
- [ ] 이전 장/다음 장 이동을 만든다.
- [ ] 글자 크기, 줄 간격, 다크모드 설정을 만든다.

### 완료 기준

- 로그인 없이 리더 화면으로 들어갈 수 있다.
- fixture 본문이 절 단위로 표시된다.
- 기본 읽기 설정이 화면에 반영된다.

## Step 2: 읽기 기록과 통독 대시보드

### 목표

local/mock storage로 마지막 위치, 읽음 완료, 통독률 흐름을 구현한다.

### 체크리스트

- [ ] 마지막 읽은 권/장/절/스크롤 위치를 저장한다.
- [ ] 이어 읽기 카드를 만든다.
- [ ] 최근 읽은 장 목록을 만든다.
- [ ] 장 단위 읽음 완료를 구현한다.
- [ ] 읽음 취소를 구현한다.
- [ ] 전체/구약/신약/권별 통독률 계산을 만든다.
- [ ] 대시보드에 통독 현황을 표시한다.
- [ ] 장별 읽음 체크 UI를 만든다.

### 완료 기준

- 새로고침 후에도 이어 읽기가 동작한다.
- 읽음 완료/취소가 통독률에 반영된다.
- mock 저장소를 DB 저장소로 바꿀 수 있는 구조가 유지된다.

## Step 3: 강조와 인용 구절 보관함

### 목표

성경 공부에 필요한 개인 기록 기능을 mock 저장소로 완성한다.

### 체크리스트

- [ ] 절 선택 상태를 만든다.
- [ ] 절 액션 메뉴를 만든다.
- [ ] 강조 색상 선택과 해제를 구현한다.
- [ ] 강조 구절 목록을 만든다.
- [ ] 강조 메모를 구현한다.
- [ ] 인용 구절 저장을 구현한다.
- [ ] 인용 구절 보관함을 만든다.
- [ ] 태그 생성과 연결을 구현한다.
- [ ] 구절 복사 기능을 구현한다.

### 완료 기준

- 강조와 인용 구절 데이터가 새로고침 후에도 유지된다.
- 리더, 강조 목록, 보관함 사이 이동이 자연스럽다.
- 구절 복사 형식이 PRD 예시와 맞는다.

## Step 4: 기본 TTS

### 목표

브라우저 TTS로 현재 장과 선택 구절을 들을 수 있게 한다.

### 체크리스트

- [ ] Web Speech API 지원 여부를 감지한다.
- [ ] 현재 장 전체 읽기를 구현한다.
- [ ] 선택 구절 읽기를 구현한다.
- [ ] 선택 구절 범위 읽기를 구현한다.
- [ ] 재생/일시정지/정지 UI를 만든다.
- [ ] 이전 절/다음 절 이동을 만든다.
- [ ] 속도 조절을 만든다.
- [ ] 현재 읽는 절 하이라이트를 만든다.

### 완료 기준

- fixture 장 전체를 TTS로 재생할 수 있다.
- 재생 중 현재 절이 표시된다.
- TTS 실패 또는 미지원 브라우저에서도 리더는 계속 동작한다.

## Step 5: MVP UX 정리와 검증

### 목표

기능을 연결하고 모바일 사용성을 정리해 v0 MVP를 시연 가능한 상태로 만든다.

### 체크리스트

- [ ] 홈에서 이어 읽기, 통독률, 최근 강조, 최근 인용 구절을 보여준다.
- [ ] 리더 하단/상단 액션 배치를 정리한다.
- [ ] 모바일 폭에서 리더와 액션 메뉴가 겹치지 않게 한다.
- [ ] 빈 상태와 오류 상태를 정리한다.
- [ ] mock 데이터 초기화 기능을 만든다.
- [ ] 핵심 시나리오를 수동 검증한다.
- [ ] 검색을 v0에 포함할지 후속으로 둘지 결정한다.

### 완료 기준

- PRD 핵심 시나리오 4개를 guest 리더와 로그인 계정 흐름으로 시연할 수 있다.
- 모바일 브라우저 기준 핵심 흐름이 깨지지 않는다.
- 남은 실제 Auth/DB/본문 삽입 작업이 명확히 분리되어 있다.

## Step 6: 실제 DB/Auth 전환 준비

### 목표

v0 MVP를 실제 운영 구조로 옮기기 위한 전환 작업을 설계한다. 이 단계는 v0 시연 필수는 아니지만, 공개 출시 전 필수다.

### 체크리스트

- [ ] 실제 인증 provider를 결정한다.
- [ ] OAuth callback, session refresh, logout 흐름을 설계한다.
- [ ] DB schema와 migration을 작성한다.
- [ ] RLS 또는 서버 권한 검증 정책을 작성한다.
- [ ] fixture repository를 DB repository로 교체한다.
- [ ] local/mock user data repository를 DB repository로 교체한다.
- [ ] 성경전서 전체 seed script를 작성한다.
- [ ] 라이선스 확인 결과에 맞춰 본문 제공 범위를 확정한다.

### 완료 기준

- mock auth를 실제 auth로 교체할 수 있다.
- fixture 본문을 DB 본문으로 교체할 수 있다.
- 개인 기록이 사용자 계정 기준으로 분리된다.

## 주요 리스크와 결정 필요 사항

- KJV 흠정역 본문 라이선스가 확정되기 전에는 전체 본문을 앱에 포함하지 않는다.
- mock Auth MVP는 실제 다기기 동기화를 보장하지 않는다.
- local/mock storage는 데이터 유실 가능성이 있으므로 운영 출시용 저장소가 아니다.
- 성경 검색은 전체 본문 DB가 들어간 뒤 품질이 결정되므로 v0에서는 후순위로 둔다.
- 공개 배포를 목표로 바꾸는 순간 실제 Auth, DB, 권한 검증, 라이선스 확인이 출시 게이트가 된다.
