# 모바일 뷰 최적화 PRD

## Summary

현재 MVP는 데스크톱 기준의 상단 탭 구조를 모바일에서도 그대로 사용한다. 모바일에서는 탭 수가 많고 홈 화면의 정보 밀도가 높아 핵심 흐름인 이어 읽기, 오늘 분량 확인, 인용 관리, 검색/설정 이동이 분산되어 보인다.

이 PRD는 기존 기능을 제거하지 않고 모바일 정보 구조만 재정리한다. 홈 탭은 세부 탭으로 나누고, 모바일 하단 네비게이션은 핵심 목적지로 제한하며, 나머지 화면은 빠른이동에서 접근하게 만든다.

## Goals

- 모바일 첫 화면에서 사용자가 즉시 이어 읽기와 오늘 통독 분량을 확인할 수 있게 한다.
- 홈 탭의 여러 컴포넌트를 세부 탭으로 분리해 긴 스크롤 의존도를 줄인다.
- 모바일 하단 네비게이션을 4개 이하의 핵심 목적지로 정리한다.
- 진척도, 강조, 검색, 설정 등 보조 화면은 빠른이동으로 접근하게 한다.
- 기존 데스크톱 탭 구조와 현재 localStorage 데이터 구조는 최대한 유지한다.
- TTS 플레이어, 구절 선택 액션 시트, 인용 저장 모달이 모바일 하단 네비게이션과 겹치지 않게 한다.

## Non-goals

- 네이티브 모바일 앱 개발
- 실제 OAuth, 운영 DB, 서버 동기화 구현
- 성경전서 전체 본문 seed
- 데이터 내보내기/가져오기
- 기존 기능 제거
- 전체 디자인 시스템 재작성
- 오프라인/PWA 캐싱

## Current Context

현재 앱의 주요 화면 키는 다음 구조를 기준으로 한다.

- `dashboard`: 홈
- `reader`: 성경
- `progress`: 진척도
- `highlights`: 강조
- `favorites`: 인용
- `search`: 검색
- `settings`: 설정

현재 모바일 개선에서 유지해야 하는 전제는 다음과 같다.

- mock auth와 localStorage 기반 개인 데이터 저장을 유지한다.
- 기존 `ViewKey` 기반 화면 전환을 우선 재사용한다.
- 이미 존재하는 빠른이동/명령 팔레트 개념을 모바일 전용 빠른이동 시트로 확장한다.
- 인용 목록, 구절 선택, TTS, 통독 플랜, 개인 노트 기능은 기존 동작을 유지한다.

## Proposed Mobile IA

### 모바일 하단 네비게이션

모바일 하단 네비게이션은 다음 4개 항목으로 제한한다.

| 항목 | 연결 화면 | 역할 |
| --- | --- | --- |
| 홈 | `dashboard` | 오늘 읽기, 통독 요약, 최근 활동의 허브 |
| 성경 | `reader` | 본문 읽기, 선택, TTS, 강조, 인용 저장 |
| 인용 | `favorites` | 인용 목록과 저장 구절 관리 |
| 빠른이동 | quick move sheet | 진척도, 강조, 검색, 설정 등 보조 화면 이동 |

`progress`, `highlights`, `search`, `settings`는 모바일 하단 네비게이션의 고정 항목에서 제외하고 빠른이동으로 접근한다.

### 홈 세부 탭

모바일 홈은 하나의 긴 대시보드 대신 내부 세부 탭으로 분리한다.

| 홈 세부 탭 | 포함 내용 | 우선순위 |
| --- | --- | --- |
| 오늘 | 이어 읽기, 오늘 통독 분량, 현재 플랜 상태, 오늘 완료 액션 | P0 |
| 통독 | 전체 통독률, 구약/신약 진행, 책별 진행 요약, 미완료 장 바로가기 | P0 |
| 활동 | 최근 읽은 위치, 최근 강조, 최근 인용, 최근 노트 | P1 |
| 공부 | 개인 노트 요약, 태그 요약, 자주 보는 인용 목록 바로가기 | P1 |

기본 진입 세부 탭은 `오늘`로 한다.

### 빠른이동

모바일 빠른이동은 전체 화면 또는 하단 시트 형태로 제공한다.

포함 목적지는 다음과 같다.

- 통독 진척도
- 강조 구절
- 검색
- 설정
- 오늘 읽을 분량
- 현재 읽는 장
- 인용 목록
- 개인 노트
- TTS 재생목록 또는 현재 재생 구간

빠른이동은 검색 입력을 포함하되, 입력 없이도 주요 목적지를 탭으로 이동할 수 있어야 한다.

## UI Behavior

### Responsive Rules

- 모바일 기준 폭은 `768px` 이하로 둔다.
- 데스크톱에서는 기존 상단 탭 네비게이션을 유지한다.
- 모바일에서는 상단 탭 목록을 숨기고 하단 네비게이션을 표시한다.
- 모바일 하단 네비게이션은 `safe-area-inset-bottom`을 고려한다.
- 하단 TTS 플레이어가 열려 있을 때 하단 네비게이션과 겹치지 않게 여백 또는 스택 순서를 조정한다.

### Home

- 홈 상단에는 현재 상태를 짧게 보여준다.
- 홈 세부 탭은 segmented control 형태로 제공한다.
- 각 세부 탭은 모바일 한 화면에서 스캔 가능한 카드/섹션 단위로 구성한다.
- 세부 탭 전환은 전체 화면 이동이 아니라 홈 내부 상태 변경으로 처리한다.
- 마지막 선택한 홈 세부 탭은 세션 중 유지한다. localStorage 저장은 선택 사항이다.

### Reader

- 모바일 리더에서는 본문 영역을 우선한다.
- 책/장 선택, 읽음 완료, TTS, 구절 선택 액션은 접히거나 시트로 정리한다.
- 다중 구절 선택 액션 시트는 하단 네비게이션 위에 표시한다.
- 인용 저장 모달은 작은 화면에서 전체 폭에 가깝게 표시하고 세로 스크롤을 허용한다.

### Favorites

- 인용 페이지는 모바일에서도 목록 선택이 먼저 보이게 한다.
- 선택한 목록의 전체 복사와 목록 삭제 버튼은 손가락 터치 영역을 충분히 확보한다.
- 구절 카드 내부의 개별 복사 기능은 유지한다.

### Quick Move

- 빠른이동 버튼을 누르면 모바일 전용 시트가 열린다.
- 시트에는 검색 입력과 목적지 목록을 함께 둔다.
- 목적지 선택 시 기존 `activeView` 전환을 사용한다.
- 빠른이동에서 `settings`로 이동한 뒤 뒤로가기 또는 닫기 흐름이 어색하지 않아야 한다.

## Data And State

이번 PRD에서 필수 DB 변경은 없다.

권장 클라이언트 상태는 다음과 같다.

```ts
type MobileHomeTab = "today" | "progress" | "activity" | "study"

type MobileQuickDestination =
  | "dashboard.today"
  | "dashboard.progress"
  | "dashboard.activity"
  | "dashboard.study"
  | "reader"
  | "progress"
  | "highlights"
  | "favorites"
  | "search"
  | "settings"
```

구현 시 주의 사항은 다음과 같다.

- 기존 `ViewKey`를 대체하지 말고 확장 상태로 둔다.
- 홈 세부 탭은 `dashboard` 안의 하위 상태로 관리한다.
- localStorage 마이그레이션은 필요할 때만 추가한다.
- 기존 저장 데이터인 진척도, 강조, 인용 목록, 노트, 설정을 변경하지 않는다.

## Implementation Phases

### Phase M-01: 모바일 IA와 상태 기반 정리

목표: 모바일 전용 네비게이션 구조를 코드에 추가하되 기존 데스크톱 구조를 깨지 않게 한다.

작업 체크리스트:

- [ ] 모바일 breakpoint 기준을 확정한다.
- [ ] `MobileHomeTab` 상태를 추가한다.
- [ ] 모바일/데스크톱 네비게이션 표시 조건을 분리한다.
- [ ] 기존 `tabs` 정의를 데스크톱 탭과 모바일 목적지로 분리한다.
- [ ] 빠른이동 목적지 목록을 구성한다.
- [ ] 데스크톱 기존 화면 전환이 유지되는지 확인한다.

수용 기준:

- [ ] 데스크톱에서는 기존 상단 탭이 유지된다.
- [ ] 모바일에서는 하단 네비게이션만 주요 네비게이션으로 보인다.
- [ ] 모든 기존 화면은 여전히 접근 가능하다.

### Phase M-02: 모바일 하단 네비게이션과 빠른이동 시트

목표: 모바일 핵심 네비게이션을 하단에 고정하고 보조 화면은 빠른이동으로 이동한다.

작업 체크리스트:

- [ ] 하단 네비게이션 컴포넌트를 추가한다.
- [ ] 하단 네비게이션 항목을 홈, 성경, 인용, 빠른이동으로 구성한다.
- [ ] 빠른이동 시트를 모바일 전용 UI로 추가한다.
- [ ] 빠른이동에서 진척도, 강조, 검색, 설정으로 이동하게 한다.
- [ ] TTS 플레이어가 있을 때 하단 여백을 조정한다.
- [ ] 모바일 브라우저 하단 safe area를 반영한다.

수용 기준:

- [ ] 375px 폭에서 하단 네비게이션 텍스트가 잘리지 않는다.
- [ ] 빠른이동으로 기존 보조 화면에 모두 접근할 수 있다.
- [ ] TTS 플레이어와 하단 네비게이션이 겹치지 않는다.

### Phase M-03: 홈 세부 탭 재구성

목표: 홈 화면을 `오늘`, `통독`, `활동`, `공부` 세부 탭으로 나눠 모바일 스캔성을 높인다.

작업 체크리스트:

- [ ] 홈 내부 segmented control을 추가한다.
- [ ] `오늘` 탭에 이어 읽기와 오늘 통독 분량을 배치한다.
- [ ] `통독` 탭에 전체 진행률과 책별 진행 요약을 배치한다.
- [ ] `활동` 탭에 최근 읽기, 강조, 인용, 노트를 배치한다.
- [ ] `공부` 탭에 태그, 노트, 자주 쓰는 인용 목록 요약을 배치한다.
- [ ] 기존 홈 컴포넌트 로직을 가능한 한 재사용한다.

수용 기준:

- [ ] 모바일 홈 첫 진입 시 `오늘` 탭이 표시된다.
- [ ] 홈 세부 탭 전환 시 페이지 전체가 불필요하게 흔들리지 않는다.
- [ ] 홈의 모든 기존 주요 정보가 세부 탭 중 하나에 남아 있다.

### Phase M-04: 리더와 인용 모바일 표면 정리

목표: 본문 읽기, 구절 선택, 인용 저장, 인용 목록 관리를 모바일에서 안정적으로 사용하게 한다.

작업 체크리스트:

- [ ] 리더 상단 조작 영역을 모바일에서 압축한다.
- [ ] 책/장 이동 UI가 작은 화면에서 넘치지 않게 한다.
- [ ] 구절 선택 액션 시트와 하단 네비게이션의 z-index/간격을 정리한다.
- [ ] 인용 저장 모달의 모바일 폭과 스크롤을 검증한다.
- [ ] 인용 페이지 목록 선택 UI를 모바일에서 우선 노출한다.
- [ ] 목록 전체 복사와 목록 삭제 버튼의 터치 영역을 조정한다.

수용 기준:

- [ ] 모바일에서 구절 선택 후 액션 시트가 화면 밖으로 나가지 않는다.
- [ ] 인용 저장 모달에서 목록 선택과 새 목록 생성이 모두 가능하다.
- [ ] 인용 페이지에서 목록 선택, 전체 복사, 삭제가 모바일에서 조작 가능하다.

### Phase M-05: 폴리시, 접근성, 브라우저 검증

목표: 모바일 최적화가 시각적으로 안정적이고 기존 흐름을 회귀시키지 않는지 확인한다.

작업 체크리스트:

- [ ] 375px, 390px, 430px, 768px 폭에서 주요 화면을 확인한다.
- [ ] 모든 터치 대상이 최소 44px에 가깝게 확보되는지 확인한다.
- [ ] 포커스 이동과 키보드 조작이 깨지지 않는지 확인한다.
- [ ] 모달/시트 오픈 시 배경 스크롤 잠금을 확인한다.
- [ ] 가로 스크롤이 생기는 화면을 제거한다.
- [ ] 데스크톱 상단 탭과 기존 레이아웃 회귀를 확인한다.

수용 기준:

- [ ] 모바일 주요 화면에서 수평 오버플로우가 없다.
- [ ] 하단 네비게이션, TTS 플레이어, 액션 시트, 모달이 서로 겹치지 않는다.
- [ ] 데스크톱 기존 탭 구조가 유지된다.
- [ ] lint, build, audit가 통과한다.

## Acceptance Criteria

- [ ] 모바일 하단 네비게이션은 4개 이하의 항목만 가진다.
- [ ] 홈 탭은 `오늘`, `통독`, `활동`, `공부` 세부 탭으로 나뉜다.
- [ ] `progress`, `highlights`, `search`, `settings`는 빠른이동으로 접근 가능하다.
- [ ] 기존 개별 화면의 기능은 제거되지 않는다.
- [ ] 375px 폭에서 네비게이션, 카드, 버튼, 모달 텍스트가 잘리지 않는다.
- [ ] 리더에서 구절 선택, TTS, 인용 저장이 모바일에서 연속 사용 가능하다.
- [ ] 인용 페이지에서 목록 선택, 목록 전체 복사, 목록 삭제가 모바일에서 동작한다.
- [ ] 새로고침 후 기존 localStorage 데이터가 유지된다.
- [ ] 데스크톱 레이아웃은 기존 상단 탭 중심으로 유지된다.

- [ ] 데스크톱에서 기존 상단 탭 이동이 정상이다.

## Risks

- 하단 네비게이션, TTS 플레이어, 구절 선택 액션 시트가 같은 하단 공간을 점유할 수 있다.
- 빠른이동으로 이동한 화면이 사용자가 찾기 어렵게 숨겨질 수 있다.
- 홈 세부 탭이 많아지면 대시보드가 다시 복잡해질 수 있다.
- 기존 `activeView` 전환 로직과 모바일 하위 탭 상태가 충돌할 수 있다.
- 모바일 시트와 모달의 z-index가 겹치면 닫기/포커스 흐름이 불안정해질 수 있다.

## Open Decisions

- 추천 하단 네비게이션은 `홈`, `성경`, `인용`, `빠른이동`이다.
- `검색`을 하단 고정 항목으로 둘지는 실제 사용 빈도 확인 후 재검토한다.
- `설정`은 빠른이동 안에 두는 것을 기본값으로 한다.
- 홈 세부 탭 선택 상태를 localStorage에 저장할지 세션 상태로만 둘지 구현 단계에서 결정한다.
- TTS 플레이어가 활성화된 경우 하단 네비게이션 위에 둘지, 미니 플레이어와 통합할지 구현 단계에서 결정한다.
