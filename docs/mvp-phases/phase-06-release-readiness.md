# Phase 6: 출시 준비

## 목표

MVP를 실제 사용자에게 공개하기 전에 라이선스, 보안, 데이터 보호, 접근성, 배포, 피드백 수집 체계를 점검한다. v0 MVP 시연은 mock Auth와 fixture 본문으로 가능하지만, 공개 출시는 실제 Auth/DB/권한 검증 전환을 게이트로 둔다.

## 구현 체크리스트

### 1. 라이선스 최종 확인

- [ ] KJV 흠정역 본문 DB 저장 가능 여부를 확인한다.
- [ ] 웹 서비스 공개 가능 여부를 확인한다.
- [ ] 검색 기능 제공 가능 여부를 확인한다.
- [ ] TTS 낭독 제공 가능 여부를 확인한다.
- [ ] 구절 복사와 공유 가능 범위를 확인한다.
- [ ] 상업적 서비스 가능 여부를 확인한다.
- [ ] 출처 표기 방식을 확인한다.
- [ ] 라이선스 확인 결과를 문서화한다.
- [ ] 권한이 불명확하면 샘플/사용자 업로드/외부 링크/본문 없는 구조 중 하나로 출시 범위를 조정한다.
- [ ] 성경전서 전체 seed는 라이선스와 DB 구현이 끝난 뒤에만 수행한다.

### 2. 보안 점검

- [ ] v0 mock 저장소와 공개 출시용 API 범위를 구분한다.
- [ ] 공개 출시용 모든 개인 데이터 API가 인증을 요구한다.
- [ ] 모든 개인 데이터 API가 사용자 소유권을 검증한다.
- [ ] Supabase 사용 시 Row Level Security를 활성화한다.
- [ ] `reading_progress` RLS 정책을 검증한다.
- [ ] `completed_chapters` RLS 정책을 검증한다.
- [ ] `highlights` RLS 정책을 검증한다.
- [ ] `favorite_verses` RLS 정책을 검증한다.
- [ ] `tags` RLS 정책을 검증한다.
- [ ] 다른 사용자의 데이터 ID를 직접 요청해도 접근이 차단되는지 테스트한다.
- [ ] 환경 변수와 서비스 키가 클라이언트 번들에 노출되지 않는지 확인한다.

### 3. 계정 및 데이터 관리

- [ ] 로그아웃 흐름을 검증한다.
- [ ] 계정 삭제 요청 흐름을 제공한다.
- [ ] 계정 삭제 시 개인 기록 삭제 범위를 정의한다.
- [ ] 사용자 설정 초기화 기능을 검토한다.
- [ ] 데이터 백업 정책을 문서화한다.
- [ ] 데이터 복구 책임 범위를 문서화한다.
- [ ] 개인정보 처리 안내 문구를 준비한다.

### 4. 품질 점검

- [ ] v0 mock 로그인 흐름을 처음부터 끝까지 검증한다.
- [ ] 성경 권/장/절 탐색 흐름을 검증한다.
- [ ] 이어 읽기 흐름을 검증한다.
- [ ] 읽음 완료와 통독률 갱신을 검증한다.
- [ ] 구절 강조와 강조 목록을 검증한다.
- [ ] 인용 구절 저장과 보관함을 검증한다.
- [ ] 구절 복사를 검증한다.
- [ ] 기본 TTS 재생을 검증한다.
- [ ] 글자 크기와 다크모드를 검증한다.
- [ ] Phase 5를 포함했다면 검색 흐름을 검증한다.

### 5. 접근성 및 반응형 점검

- [ ] 모바일 브라우저에서 홈/대시보드가 깨지지 않는다.
- [ ] 모바일 브라우저에서 성경 리더가 읽기 편하다.
- [ ] 주요 버튼 터치 영역이 충분하다.
- [ ] 키보드 탐색으로 핵심 화면을 사용할 수 있다.
- [ ] 색상 대비가 충분하다.
- [ ] 강조 색상 외에도 선택/상태를 알 수 있다.
- [ ] 글자 크기 확대 후 레이아웃이 무너지지 않는다.
- [ ] TTS 컨트롤에 접근 가능한 이름이 있다.

### 6. 성능 점검

- [ ] 성경 장 로딩 1초 이내 목표를 확인한다.
- [ ] 대시보드 로딩 2초 이내 목표를 확인한다.
- [ ] 검색 포함 시 검색 결과 2초 이내 목표를 확인한다.
- [ ] 초기 번들 크기를 점검한다.
- [ ] 느린 네트워크에서 리더 기본 사용성을 확인한다.
- [ ] 에러 상태와 재시도 상태를 확인한다.

### 7. 배포 준비

- [ ] Vercel 프로젝트를 연결한다.
- [ ] Supabase 프로젝트를 연결한다.
- [ ] 운영 환경 변수를 설정한다.
- [ ] 데이터베이스 마이그레이션 적용 절차를 정리한다.
- [ ] 배포 전 빌드가 성공하는지 확인한다.
- [ ] 운영 도메인과 HTTPS를 확인한다.
- [ ] 오류 로깅 또는 최소한의 운영 로그 확인 방법을 마련한다.
- [ ] 배포 후 핵심 사용자 시나리오를 운영 환경에서 다시 검증한다.
- [ ] 공개 출시가 아니라 내부 시연 배포라면 mock Auth와 fixture 본문 사용을 화면 또는 문서에 명시한다.

### 8. 성공 지표와 피드백

- [ ] 일간 활성 사용자 수 측정 방식을 정한다.
- [ ] 주간 활성 사용자 수 측정 방식을 정한다.
- [ ] 평균 읽은 장 수 측정 방식을 정한다.
- [ ] 평균 세션 시간 측정 방식을 정한다.
- [ ] 통독 완료율 측정 방식을 정한다.
- [ ] 구절 강조 수를 측정한다.
- [ ] 인용 구절 저장 수를 측정한다.
- [ ] TTS 재생 횟수를 측정한다.
- [ ] 구절 복사 횟수를 측정한다.
- [ ] 사용자 피드백 수집 채널을 만든다.

## 수용 기준

- 본문 라이선스 리스크가 출시 범위에 반영되어 있다.
- v0 시연은 mock 사용자 기준으로 동작하고, 공개 출시는 개인 데이터 접근이 계정별로 제한된다.
- 운영 환경에서 핵심 사용자 시나리오가 성공한다.
- 모바일 브라우저에서 핵심 기능을 사용할 수 있다.
- 배포, 롤백, 피드백 수집 방식이 정리되어 있다.
