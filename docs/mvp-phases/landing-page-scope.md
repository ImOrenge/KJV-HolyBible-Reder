# 랜딩페이지 구조 스코프

## Summary

KJV 리더노트의 첫 페이지를 공개 랜딩 페이지로 전환한다. 랜딩의 메시지는 단순 기능 소개가 아니라 "읽던 자리, 붙잡은 구절, 통독 흐름이 흩어지지 않는 개인 성경 공부 공간"이라는 제품 약속을 중심에 둔다.

랜딩은 방문자가 "이 앱은 성경 읽기 앱이 아니라 내 통독과 구절 기록을 이어주는 도구"라고 느끼게 해야 한다. 기능명은 보조 정보로 두고, 첫 화면에서는 사용자가 이미 겪는 문제와 앱이 주는 변화가 먼저 보여야 한다.

현재 `src/app/page.tsx`는 로그인 사용자를 확인한 뒤 앱 본 화면을 렌더링하고, 비로그인 사용자는 `/auth/login`으로 보낸다. 랜딩 도입 후에는 `/`를 공개 랜딩으로 바꾸고, 실제 앱 진입점은 `/app` 같은 보호 경로로 분리한다.

확인일: 2026-06-23

## Goals

- 비로그인 사용자가 앱의 목적과 사용 이유를 5초 안에 이해하게 한다.
- 첫 화면에서 "내 성경 통독과 구절 기록을 잃어버리지 않는 개인 공부 공간"이라는 정체성을 명확히 보여준다.
- 기능 목록보다 사용자의 실제 고민을 먼저 건드린다.
- 로그인하면 읽기 위치, 강조, 인용 구절, 설정이 계정 기준으로 분리된다는 점을 설명한다.
- 로그인과 계정 만들기 CTA를 "기록을 이어가기 위한 다음 행동"으로 배치한다.
- 실제 앱 UI preview를 사용해 기능을 과장 없이 보여준다.
- 추후 Supabase Auth, DB repository, RLS 전환과 충돌하지 않는 route 구조를 잡는다.

## Non-goals

- Google, GitHub, Apple 등 외부 OAuth provider SSO 구현
- 가격 정책, 결제, 구독 플랜
- 커뮤니티, 공유 피드, 공개 프로필
- 고품질 AI 음성, 오프라인 모드, 구절 이미지 생성
- 성경전서 전체 본문 라이선스 확정 전 전체 본문 홍보
- 실제 개인 데이터 DB 동기화가 완료되지 않은 상태에서 "모든 기기 동기화"라고 표현하는 것

### Copy Principles

- 기능을 먼저 말하지 말고, 사용자가 얻는 변화부터 말한다.
- "저장", "동기화", "RLS" 같은 기술어는 랜딩 본문에서는 뒤로 보낸다.
- "성경 공부를 더 잘하세요"처럼 추상적인 문구보다 "읽던 자리", "붙잡은 구절", "다시 꺼내 쓸 구절"처럼 손에 잡히는 표현을 쓴다.
- 보안과 인증은 PR 문구로 과장하지 않고, "내 기록을 내 계정 기준으로 분리"하는 설명으로 둔다.
- 전체 본문, 다기기 동기화, 외부 SSO는 구현 완료 전까지 약속처럼 쓰지 않는다.

| 항목 | 현재 상태 | 랜딩 반영 원칙 |
| --- | --- | --- |
| 앱 이름 | `KJV 리더노트` | H1과 nav brand에 그대로 사용 |
| 핵심 기능 | 이어 읽기, 통독률, 강조, 인용 구절, TTS, 설정 | 이점 섹션에서 사용자 변화의 근거로 사용 |

| SSO | 외부 provider SSO는 현재 범위 아님 | 사용자용 문구는 "계정 기반 로그인 구조"로 표현 |
| 개인 데이터 | 사용자 ID별 localStorage 분리, DB/RLS 전환 예정 | "계정별 분리"는 말하되 "다기기 동기화 완료"는 말하지 않음 |
| 성경 본문 | 공개 KJV/fixture/DB 전환 경계가 있음 | 전체 본문 제공 범위는 라이선스 게이트 이후 확정 |

## Target Route IA

| Route | 접근 | 역할 |
| --- | --- | --- |
| `/` | 공개 | 랜딩 페이지 |
| `/app` | 로그인 필요 | 실제 KJV 리더노트 앱 |
| `/auth/login` | 공개 | 이메일 로그인 |
| `/auth/sign-up` | 공개 | 계정 만들기 |
| `/auth/reset-password` | 공개 | 비밀번호 재설정 |
| `/auth/update-password` | 세션 필요 | 새 비밀번호 저장 |

### Route Behavior

- 비로그인 사용자가 `/`에 오면 랜딩을 본다.
- 비로그인 사용자가 `/app`에 오면 `/auth/login?next=/app` 또는 `/auth/login`으로 이동한다.
- 로그인 사용자가 `/`에 오면 랜딩을 그대로 볼 수 있되, CTA는 `내 리더노트 열기`로 바뀐다.
- 로그인 사용자가 `/auth/login` 또는 `/auth/sign-up`에 오면 `/app`으로 이동한다.
- 로그인 후 기본 redirect는 `/app`으로 둔다.

### 3. Product Preview

목표:

- 실제 앱의 대시보드, 리더, 강조/인용 흐름을 보여준다.
- 과장된 마케팅 이미지 대신 현재 구현된 UI를 기반으로 만든 preview를 사용한다.
- preview는 단순 스크린샷이 아니라 "읽고, 표시하고, 다시 여는" 흐름을 말해야 한다.

구성:

| Dashboard | 이어 읽기 카드, 통독률, 최근 활동 | 오늘 어디서 다시 시작할지 바로 보입니다. |
| Reader | 권/장/절 본문, 선택 구절 상태 | 읽다가 붙잡은 구절을 그 자리에서 남깁니다. |
| Study tools | 강조 색상, 메모, 인용 구절 저장 | 나중에 다시 쓸 구절을 읽는 흐름에서 저장합니다. |
| TTS | 재생/일시정지, 속도, 현재 절 표시 | 눈으로 읽기 어려운 시간에도 이어 들을 수 있습니다. |

구현 원칙:

- 랜딩용 static preview data를 사용한다.
- 개인 데이터처럼 보이는 정보는 fixture임을 코드 상수로 분리한다.
- preview가 실제 로그인 세션이나 localStorage를 읽지 않게 한다.

목표:

- 사용자가 로그인 전에 "내 공부에 어떤 변화가 생기는지" 판단하게 한다.
- 기능명은 제목보다 설명 안쪽에 배치한다.

이점 카드:

| 연결 기능 | 제목 | 설명 |
| --- | --- | --- |
| 이어 읽기 | 다시 시작할 곳을 찾느라 멈추지 않습니다. | 마지막으로 읽던 권, 장, 절을 기억해 다음 방문에서 바로 이어갑니다. |
| 통독률 | 통독이 감이 아니라 기록으로 보입니다. | 전체, 구약/신약, 권별 진행을 숫자와 화면 흐름으로 확인합니다. |
| 하이라이트 | 지나가면 잊히는 구절을 붙잡아 둡니다. | 중요한 구절을 색상과 메모로 표시해 다시 볼 수 있게 남깁니다. |
| 인용 보관함 | 다시 쓸 구절을 따로 찾지 않아도 됩니다. | 설교, 묵상, 글쓰기, SNS에 자주 쓰는 구절을 보관합니다. |
| TTS | 읽기 어려운 시간에도 흐름을 이어갑니다. | 현재 장이나 선택 구절을 브라우저 음성으로 들을 수 있습니다. |

### 5. Account Structure

목표:

- 사용자가 왜 로그인해야 하는지 납득하게 한다.
- "SSO"를 기술 내부 용어로만 쓰지 않고, 사용자에게는 계정 기반 저장 구조로 풀어 설명한다.

권장 사용자 문구:

```txt
로그인은 단순한 출입문이 아니라
내 성경 공부 기록을 내 계정 기준으로 이어가기 위한 기준입니다.

읽던 위치, 강조한 구절, 인용 보관함, 읽기 설정이
다른 사람의 기록과 섞이지 않도록 사용자 ID별로 분리됩니다.
```

구조 설명:

```mermaid
flowchart LR
  A["사용자"] --> B["이메일 로그인"]
  B --> C["Supabase Auth"]
  C --> D["세션 쿠키"]
  D --> E["/app"]
  E --> F["사용자 ID별 개인 데이터"]
  F --> G["향후 DB + RLS 동기화"]
```

표현 기준:

| 표현 | 사용 여부 | 이유 |
| --- | --- | --- |
| 이메일 로그인 | 사용 | 현재 구현과 일치 |
| 계정 기반 저장 | 사용 | 사용자에게 이해 쉬움 |
| SSO-ready | 제한 사용 | 외부 provider가 정해지기 전에는 내부 문서에서만 사용 |
| Google로 계속하기 | 사용 금지 | 현재 non-goal |
| 모든 기기에서 자동 동기화 | 사용 금지 | DB repository 전환 완료 전 과장 |

### 6. Study Workflow

목표:

- 기능을 나열하는 데서 끝내지 않고 사용자의 하루 공부 흐름을 보여준다.

흐름:

| 단계 | 제목 | 설명 |
| --- | --- | --- |

## Visual Design Direction

- 랜딩은 문서형 SaaS가 아니라 실제 공부 도구의 첫 화면처럼 설계한다.
- 큰 장식 카드보다 실제 앱 preview와 기능 밀도를 우선한다.
- Hero는 앱 이름과 실제 리더/대시보드 preview가 첫 viewport에서 보이게 한다.
- 색상은 고대 문헌 느낌의 갈색/베이지 일변도를 피하고, 흰 배경, 짙은 텍스트, 절제된 녹색 또는 금색 accent를 사용한다.
- 모바일에서는 CTA를 상단과 하단에 반복하되, 기능 설명 텍스트가 버튼이나 preview를 가리지 않게 한다.
- 랜딩에서 설명용 텍스트가 과해지면 로그인 전환이 약해지므로 각 섹션은 짧은 제목, 한 문장 설명, 실제 UI 단서 중심으로 둔다.

## Implementation Phases

### L-01: Route Split

작업:

- [ ] 현재 `src/app/page.tsx`의 보호 앱 렌더링을 `/app` 경로로 이동한다.
- [ ] `/`는 공개 랜딩 페이지로 바꾼다.
- [ ] `/app`에서 Supabase user가 없으면 로그인으로 보낸다.
- [ ] 로그인 성공 redirect 기본값을 `/app`으로 정리한다.

수용 기준:

- [ ] 비로그인 사용자가 `/`에서 랜딩을 볼 수 있다.
- [ ] 비로그인 사용자가 `/app`에 접근하면 로그인으로 이동한다.
- [ ] 로그인 사용자는 `/app`에서 기존 앱을 그대로 사용할 수 있다.

### L-02: Landing Content Components

작업:

- [ ] 랜딩 전용 component 또는 route component를 만든다.
- [ ] Hero, preview, 기능, 로그인 구조, 데이터 보호, final CTA 섹션을 구현한다.
- [ ] CTA 링크를 `/auth/login`, `/auth/sign-up`, `/app`으로 연결한다.
- [ ] 로그인 상태에 따라 CTA 문구를 바꿀지 결정한다.

수용 기준:

- [ ] 첫 viewport에서 앱 이름, 제품 약속, 로그인 CTA, 제품 preview가 보인다.
- [ ] Hero copy가 기능 나열이 아니라 사용자 문제와 제품 변화를 말한다.
- [ ] 랜딩에 현재 구현되지 않은 OAuth provider나 동기화 약속이 없다.
- [ ] 모든 CTA가 올바른 route로 이동한다.

### L-03: Responsive And Accessibility

작업:

- [ ] 모바일 390px 폭에서 hero, CTA, preview가 겹치지 않게 한다.
- [ ] desktop 1440px 폭에서 preview가 과도하게 늘어나지 않게 한다.
- [ ] CTA 버튼에 명확한 accessible name을 둔다.
- [ ] 섹션 heading hierarchy를 `h1`, `h2`, `h3` 순서로 유지한다.

수용 기준:

- [ ] 모바일과 데스크톱에서 텍스트가 버튼 또는 preview 위에 겹치지 않는다.
- [ ] 키보드 탭 순서가 nav, CTA, 섹션 링크 순서로 자연스럽다.
- [ ] 컬러 대비가 본문과 CTA에서 충분하다.

### L-04: Verification

작업:

- [ ] `npm run lint`
- [ ] `npm run build`
- [ ] 브라우저에서 `/`, `/app`, `/auth/login`, `/auth/sign-up` 흐름 확인
- [ ] 모바일 viewport screenshot 확인
- [ ] 로그인 상태와 비로그인 상태 CTA 확인

수용 기준:

- [ ] `/`는 공개 접근 가능하다.
- [ ] `/app`는 보호된다.
- [ ] 로그인 후 기존 리더, 통독, 강조, 인용, TTS 흐름이 깨지지 않는다.
- [ ] 랜딩 카피가 현재 구현 범위를 과장하지 않는다.

## Acceptance Criteria

- [ ] 랜딩 첫 화면에 `KJV 리더노트`가 주요 신호로 보인다.
- [ ] 랜딩 첫 화면이 "읽던 자리, 붙잡은 구절, 통독 흐름을 이어주는 도구"로 읽힌다.
- [ ] 사용자가 앱의 핵심 이점을 로그인 전에 이해할 수 있다.
- [ ] 카피가 기능 나열에 그치지 않고 사용자 문제와 제품 변화를 말한다.
- [ ] 로그인 필요성이 "개인 공부 기록을 계정 기준으로 분리하기 위해서"로 설명된다.
- [ ] SSO/인증 구조는 현재 구현과 맞게 Supabase 이메일 로그인 중심으로 설명된다.
- [ ] 외부 OAuth provider, 다기기 동기화, 전체 본문 제공을 완료 기능처럼 표현하지 않는다.
- [ ] `/`와 `/app`의 공개/보호 경계가 분리된다.
- [ ] 로그인 후 기존 앱 기능이 회귀하지 않는다.
- [ ] 모바일과 데스크톱에서 시각적으로 검증된 screenshot을 남긴다.

## Open Decisions

- 외부 provider SSO를 실제로 넣을지, 이메일 로그인만 유지할지 결정해야 한다.
- 로그인한 사용자가 `/` 방문 시 자동으로 `/app`으로 보낼지, 랜딩을 보여주고 CTA만 바꿀지 결정해야 한다. 권장은 랜딩 유지 후 CTA 변경이다.
- preview를 실제 앱 component 일부를 재사용해 만들지, 랜딩 전용 static preview로 만들지 결정해야 한다. 권장은 static preview다.
- 로컬 데이터 가져오기 안내를 랜딩에서 언급할지, 로그인 후 앱 내부 안내로만 둘지 결정해야 한다. 권장은 앱 내부 안내다.
